# Продвинутые техники и технологии работы с ИИ

**Дата фиксации:** 13 июля 2026  
**Назначение файла:** расширение знаний для дальнейшего развития курса и личной практики.

Этот документ не привязан к юридическому курсу. Это обзор самых сильных современных подходов к работе с ИИ: от продвинутого промптинга до агентных систем, RAG, evals, MCP и автоматической оптимизации.

## 1. Agentic Workflows

**Суть:** ИИ используется не как один чат, а как система ролей, шагов, инструментов и проверок.

Типовые паттерны:

- `planner -> executor -> critic`;
- `researcher -> verifier -> writer`;
- `supervisor agent -> specialist agents`;
- `human-in-the-loop`, когда человек утверждает рискованные действия;
- `agent as tool`, когда один агент вызывает другого как специализированный инструмент.

Что это даёт:

- сложная задача делится на этапы;
- разные роли не смешиваются;
- проверка отделяется от генерации;
- можно строить повторяемые рабочие процессы.

Инструменты:

- OpenAI Agents SDK: https://openai.github.io/openai-agents-python/
- LangGraph: https://docs.langchain.com/oss/python/langgraph/overview
- CrewAI: https://docs.crewai.com/
- AutoGen: https://microsoft.github.io/autogen/stable/

## 2. Multi-Agent Systems

**Суть:** несколько агентов взаимодействуют между собой: спорят, проверяют, делегируют, передают результаты.

Примеры ролей:

- координатор;
- исследователь;
- аналитик;
- критик;
- редактор;
- проверяющий;
- инструментальный агент, который умеет работать с файлами, кодом или браузером.

Продвинутые паттерны:

- **handoff**: агент передаёт задачу более подходящему агенту;
- **manager pattern**: центральный агент управляет специалистами;
- **swarm pattern**: несколько агентов работают более свободно;
- **debate pattern**: агенты спорят и затем выбирается лучший вывод;
- **reflection pattern**: агент критикует и улучшает свой же ответ.

Риск: многоагентность легко создаёт иллюзию надёжности. Если три агента ошиблись одинаково, это не доказательство.

## 3. Tool Calling / Function Calling

**Суть:** модель не только пишет текст, но и вызывает инструменты.

Примеры инструментов:

- поиск;
- калькулятор;
- база данных;
- OCR;
- календарь;
- CRM;
- генератор документов;
- API компании;
- локальная функция проверки;
- скрипт обработки таблицы.

Пример идеи:

```text
Пользователь спрашивает: "Сколько мы потратили на аренду за квартал?"

Модель:
1. понимает задачу;
2. вызывает функцию чтения таблицы;
3. вызывает функцию фильтрации периода;
4. считает сумму;
5. возвращает объяснение и результат.
```

Ценность: ИИ превращается из советчика в управляющий слой над инструментами.

## 4. Structured Outputs

**Суть:** модель возвращает не свободный текст, а строгую структуру.

Пример:

```json
{
  "risk": "Одностороннее изменение арендной платы",
  "source_quote": "Арендодатель вправе изменить размер арендной платы...",
  "confidence": "medium",
  "needs_human_review": true,
  "recommended_action": "Проверить допустимость условия и запросить уточнение"
}
```

Зачем это нужно:

- результат можно автоматически проверять;
- его можно класть в таблицу или базу;
- проще строить пайплайны;
- меньше неоднозначности;
- легче делать evals.

Технически это реализуется через JSON Schema, Pydantic-модели, function calling, typed outputs.

## 5. RAG

**RAG** — Retrieval-Augmented Generation. Модель отвечает не только из своих знаний, а получает релевантные фрагменты из базы документов.

Базовый процесс:

1. Документы разбиваются на фрагменты.
2. Фрагменты индексируются.
3. По вопросу ищутся релевантные куски.
4. Модель получает найденный контекст.
5. Ответ строится на основе источников.

Продвинутые элементы:

- chunking strategy;
- embeddings;
- vector search;
- keyword search;
- hybrid search;
- reranking;
- metadata filters;
- citation grounding;
- query rewriting;
- multi-hop retrieval.

Инструменты:

- LlamaIndex: https://developers.llamaindex.ai/
- Haystack: https://docs.haystack.deepset.ai/
- LangChain: https://docs.langchain.com/
- Qdrant: https://qdrant.tech/
- Weaviate: https://weaviate.io/
- Pinecone: https://www.pinecone.io/

## 6. GraphRAG

**Суть:** поверх документов строится граф сущностей и связей.

Обычный RAG хорошо ищет фрагменты. GraphRAG помогает отвечать на вопросы, где важны связи:

- кто с кем связан;
- какие события следуют друг за другом;
- какие документы относятся к одному делу;
- какие сущности повторяются в разных источниках.

Типовая архитектура:

```text
документы -> извлечение сущностей -> граф связей -> поиск по графу + поиск по тексту -> ответ с источниками
```

Инструменты:

- Neo4j;
- LlamaIndex GraphRAG;
- LangChain + graph stores;
- Microsoft GraphRAG-подходы.

## 7. MCP

**MCP** — Model Context Protocol.

Суть: стандартный способ подключать ИИ-приложения к внешним данным и инструментам. Его часто описывают как «USB-C для ИИ».

Что может подключаться через MCP:

- локальные файлы;
- базы данных;
- GitHub;
- Figma;
- Google Calendar;
- Notion-подобные базы;
- поисковые инструменты;
- корпоративные API;
- специализированные рабочие процессы.

Зачем это важно:

- меньше кастомных интеграций;
- один сервер может работать с разными ИИ-клиентами;
- агент получает доступ к реальным данным и действиям;
- проще строить расширяемые рабочие среды.

Официальная документация: https://modelcontextprotocol.io/docs/getting-started/intro

Риск: MCP усиливает возможности ИИ, но также повышает требования к безопасности: права доступа, prompt injection, утечки данных, лишние действия.

## 8. Evals

**Суть:** качество ИИ-системы измеряется тестами, а не ощущениями.

Примеры evals:

- 30 типовых задач для проверки промпта;
- сравнение моделей на одном наборе кейсов;
- проверка галлюцинаций;
- проверка полноты извлечения;
- проверка соблюдения формата;
- проверка устойчивости к вредным или странным запросам.

Пример простой метрики:

```text
Задача: извлечь 10 условий из договора.

Метрики:
- найдено условий: 8/10;
- ложных находок: 2;
- цитаты есть: 7/8;
- юрисдикция не смешана: да;
- формат соблюдён: да.
```

Инструменты:

- OpenAI Evals: https://platform.openai.com/docs/guides/evals
- promptfoo: https://www.promptfoo.dev/docs/intro/
- LangSmith: https://docs.smith.langchain.com/
- Weights & Biases Weave: https://weave-docs.wandb.ai/

## 9. DSPy

**Суть:** переход от ручного промптинга к программированию ИИ-пайплайнов.

Идея DSPy: **program, don’t prompt**.

Вместо ручного текста промпта задаются:

- входы;
- выходы;
- модуль рассуждения;
- примеры;
- метрика качества.

Затем система может автоматически улучшать инструкции и демонстрации под выбранную метрику.

Где полезно:

- классификация;
- извлечение данных;
- RAG;
- fact-checking;
- повторяемые корпоративные задачи;
- оптимизация промптов под дешёвую модель.

Официальный сайт: https://dspy.ai/

## 10. Prompt Optimization

**Суть:** промпты улучшаются системно, а не вручную на глаз.

Подходы:

- A/B-тестирование промптов;
- few-shot подбор примеров;
- автоматическая генерация вариантов;
- self-refinement;
- optimizer по метрике;
- перенос поведения сильной модели на более дешёвую.

Пример процесса:

```text
1. Собрать 50 типовых задач.
2. Определить критерии качества.
3. Прогнать базовый промпт.
4. Создать 5 вариантов промпта.
5. Сравнить по метрикам.
6. Оставить лучший.
7. Повторить через месяц.
```

## 11. Human-in-the-loop

**Суть:** человек остаётся в цепочке там, где ошибка дорогая.

Примеры:

- ИИ готовит письмо, человек отправляет;
- ИИ предлагает правку договора, человек утверждает;
- ИИ готовит команду, человек запускает;
- ИИ нашёл риск, человек квалифицирует;
- ИИ собрал отчёт, человек проверяет источники.

Уровни контроля:

- approve / reject;
- edit before continue;
- ask human if confidence low;
- require confirmation before external action;
- require source citation before final answer.

Это ключевой паттерн для юридических, финансовых, медицинских и корпоративных задач.

## 12. Guardrails

**Суть:** ограничения и проверки, которые не дают ИИ-системе делать опасные или некачественные вещи.

Виды guardrails:

- проверка входа;
- проверка выхода;
- запрет на определённые действия;
- валидация JSON;
- проверка ссылок;
- проверка токсичности;
- проверка утечки секретов;
- проверка prompt injection;
- ограничение инструментов по правам.

Пример:

```text
Если ответ содержит правовой вывод без цитаты или ссылки на проверку,
верни ошибку и попроси модель переписать ответ.
```

## 13. Red Teaming

**Суть:** намеренная проверка ИИ-системы на поломки.

Что проверяют:

- jailbreak;
- prompt injection;
- утечки системных инструкций;
- утечки пользовательских данных;
- вредные советы;
- обход ограничений;
- некорректные вызовы инструментов;
- ложную уверенность;
- отказ там, где ответ допустим;
- согласие там, где нужен отказ.

Инструменты:

- promptfoo red teaming;
- Garak;
- Lakera;
- Protect AI;
- встроенные guardrails и evals в агентных фреймворках.

## 14. Model Routing

**Суть:** выбирать модель под задачу, а не использовать одну модель для всего.

Примеры маршрутизации:

- дешёвая модель для классификации;
- сильная модель для сложного анализа;
- длинноконтекстная модель для больших документов;
- локальная модель для приватных данных;
- мультимодальная модель для сканов и изображений;
- специализированная модель для кода.

Паттерн:

```text
Если задача простая -> дешёвая модель.
Если есть риск ошибки -> сильная модель + проверка.
Если данные чувствительные -> локальный контур.
Если документ длинный -> модель с большим контекстом или RAG.
```

## 15. Model Cascades

**Суть:** задача проходит через несколько уровней моделей.

Пример:

```text
1. Маленькая модель классифицирует задачу.
2. Средняя модель извлекает данные.
3. Сильная модель делает финальный анализ.
4. Проверяющая модель ищет ошибки.
```

Зачем:

- дешевле;
- быстрее;
- надёжнее;
- можно выделять сильную модель только на сложные шаги.

## 16. Fine-tuning

**Суть:** модель дообучается на своих примерах.

Полезно, когда есть:

- много однотипных задач;
- стабильный формат ответа;
- накопленные хорошие примеры;
- потребность в специфическом стиле;
- классификация или извлечение данных.

Не всегда нужно для:

- редких задач;
- задач, где достаточно RAG;
- задач, где меняется правовая или фактическая база;
- ситуаций, где проблема в плохом процессе, а не в модели.

## 17. Distillation

**Суть:** поведение сильной дорогой модели переносится в более дешёвую модель.

Пример:

```text
Сильная модель генерирует хорошие ответы на 1000 задач.
Эти ответы используются как обучающие примеры.
Меньшая модель учится повторять нужный стиль и структуру.
```

Ценность:

- снижение стоимости;
- ускорение;
- стандартизация ответа;
- возможность запускать более простую модель в массовых сценариях.

## 18. Synthetic Data

**Суть:** ИИ создаёт тренировочные данные, тестовые кейсы, edge cases и плохие примеры.

Применения:

- создать 100 вариантов договорных пунктов;
- сгенерировать ошибки для тренажёра;
- подготовить набор для evals;
- создать примеры для fine-tuning;
- сгенерировать сложные негативные кейсы;
- проверить устойчивость промпта.

Важно: синтетические данные нужно проверять. ИИ может сгенерировать правдоподобный мусор.

## 19. Multimodal AI

**Суть:** работа не только с текстом, но и с изображениями, PDF, таблицами, аудио, видео.

Примеры:

- анализ скана документа;
- извлечение данных из PDF;
- чтение таблиц;
- описание графиков;
- анализ скриншотов интерфейса;
- голосовой ассистент;
- работа с презентациями;
- видеоразбор.

Инструменты и направления:

- GPT-4o / GPT-5-class multimodal models;
- Gemini;
- Claude с визуальным анализом;
- LlamaParse;
- OCR + LLM;
- vision RAG.

## 20. Computer Use / Browser Agents

**Суть:** ИИ управляет браузером или компьютером.

Что может делать:

- открыть сайт;
- заполнить форму;
- скачать файл;
- сравнить страницы;
- собрать данные;
- работать с веб-приложением;
- повторять рутинный процесс.

Риск: ошибка превращается не просто в плохой текст, а в неправильное действие. Нужны права, лимиты и подтверждения.

## 21. Long-Context Engineering

**Суть:** работа с очень большими объёмами текста.

Техники:

- map-reduce summarization;
- hierarchical summaries;
- rolling memory;
- section-by-section extraction;
- summary with citations;
- context compression;
- retrieval over chunks;
- long-context evals;
- контроль потери середины документа.

Главное правило: большое контекстное окно не отменяет методику работы с длинными документами.

## 22. Memory Systems

**Суть:** ИИ-система хранит рабочую и долгосрочную память.

Типы памяти:

- краткосрочная память внутри задачи;
- долговременная память пользователя;
- память проекта;
- векторная память;
- структурированная память;
- эпизодическая память прошлых решений.

Пример:

```text
После каждого шага обновляй:
1. известные факты;
2. открытые вопросы;
3. принятые допущения;
4. решения пользователя;
5. следующие действия.
```

## 23. Observability

**Суть:** трассировка и мониторинг ИИ-систем.

Что отслеживают:

- какой промпт ушёл;
- какой контекст был найден;
- какие инструменты вызваны;
- сколько стоил запрос;
- где агент зациклился;
- какой шаг дал ошибку;
- какие ответы ухудшились после обновления.

Инструменты:

- LangSmith;
- OpenAI Agents SDK tracing;
- Weights & Biases Weave;
- Arize Phoenix;
- OpenTelemetry.

## 24. AI Security

**Суть:** безопасность ИИ-систем как отдельная дисциплина.

Ключевые угрозы:

- prompt injection;
- data exfiltration;
- tool poisoning;
- malicious documents;
- supply chain risk в MCP / plugins;
- over-permissioned agents;
- jailbreaks;
- indirect prompt injection через веб-страницы и документы;
- небезопасная память;
- утечка секретов в логи.

Базовые меры:

- минимальные права;
- allowlist инструментов;
- подтверждение действий;
- проверка источников;
- изоляция окружений;
- логирование;
- red teaming;
- регулярные evals.

## 28. Главная мысль

Продвинутая работа с ИИ — это не знание «секретных промптов».

Это умение проектировать систему:

```text
данные -> контекст -> модель -> инструменты -> проверка -> человек -> артефакт
```

Чем лучше человек управляет этим процессом, тем меньше он зависит от магии конкретной модели и тем больше получает от любой новой технологии.
